Được. Với định hướng Focus-Node / “Ứng dụng quản lý tri thức và hỗ trợ học tập cá nhân”, mình sẽ không vội nhảy ngay vào viết `CREATE TABLE`. Để database đạt **3NF**, đủ lớn, dễ mở rộng và đúng nghiệp vụ, ta cần đi theo hướng:

> **Nghiên cứu nghiệp vụ → xác định module → xác định thực thể → xác định quan hệ → chuẩn hóa 3NF → thiết kế ERD → viết SQL Server schema → kiểm thử bằng dữ liệu mẫu và truy vấn thực tế.**
> 

Dưới đây là **kế hoạch nghiên cứu và thiết kế cơ sở dữ liệu SQL Server** cho ứng dụng của bạn.

---

# 1. Mục tiêu của database

Database cần phục vụ cho một ứng dụng học tập cá nhân có các chức năng chính:

1. Quản lý người dùng.
2. Quản lý ghi chú / tri thức cá nhân.
3. Quản lý tag, chủ đề, môn học, tài liệu.
4. Quản lý nhiệm vụ học tập.
5. Quản lý phiên tập trung / Pomodoro.
6. Quản lý roadmap học tập.
7. Quản lý ôn tập / flashcard / câu hỏi.
8. Quản lý học nhóm LAN.
9. Quản lý file chia sẻ.
10. Quản lý thống kê học tập.
11. Quản lý cài đặt cá nhân.
12. Quản lý thông báo / nhắc nhở.

Database phải đạt:

- **1NF**: dữ liệu nguyên tử, không lưu danh sách trong một cột.
- **2NF**: mọi thuộc tính không khóa phải phụ thuộc đầy đủ vào khóa chính.
- **3NF**: không có phụ thuộc bắc cầu giữa các thuộc tính không khóa.
- Hạn chế trùng lặp dữ liệu.
- Dễ mở rộng module.
- Phù hợp SQL Server.
- Có khóa chính, khóa ngoại, unique constraint, check constraint hợp lý.
- Không lưu dữ liệu thống kê tổng hợp nếu có thể tính từ bảng gốc, trừ khi cần cache/performance.

# 2. Tư duy thiết kế tổng thể

Ứng dụng của bạn nên xoay quanh vòng lặp học tập:

```
Capture → Organize → Plan → Focus → Review → Reflect
```

Từ vòng lặp này, database cũng nên chia thành các vùng nghiệp vụ:

| Vòng lặp | Ý nghĩa | Nhóm bảng tương ứng |
| --- | --- | --- |
| Capture | Ghi lại kiến thức | Notes, Quick Notes, Files |
| Organize | Tổ chức tri thức | Tags, Categories, Subjects, Note Links |
| Plan | Lập kế hoạch học | Tasks, Roadmaps, Goals |
| Focus | Tập trung học | Pomodoro, Focus Sessions |
| Review | Ôn tập | Flashcards, Review Schedules, Quizzes |
| Reflect | Nhìn lại tiến bộ | Analytics Views, Study Logs |
| Collaborate | Học cùng nhau | LAN Rooms, Members, File Transfers |

# 4. Kế hoạch nghiên cứu database

## Giai đoạn 1: Nghiên cứu nghiệp vụ cốt lõi

### Mục tiêu

Hiểu rõ ứng dụng đang giải quyết vấn đề gì và dữ liệu nào cần lưu.

### Câu hỏi cần trả lời

#### Về người dùng

- Người dùng là ai?
- Một người dùng có nhiều workspace không?
- Có cần đăng nhập không?
- Một người dùng có thể tham gia nhiều phòng học không?
- Người dùng có cài đặt Pomodoro riêng không?
- Người dùng có mục tiêu học tập riêng không?

#### Về tri thức

- Một note thuộc một môn học hay nhiều môn học?
- Một note có thể có nhiều tag không?
- Một note có thể liên kết với note khác không?
- Note có trạng thái học tập không?
- Note có cần lưu lịch sử chỉnh sửa không?
- Quick note có được chuyển thành note chính thức không?

#### Về task

- Task có thể gắn với note không?
- Task có thể thuộc roadmap không?
- Task có deadline không?
- Task có priority không?
- Task có estimated time và actual time không?
- Task có subtasks không?

#### Về Pomodoro

- Một focus session gắn với task hay note?
- Một session có thể không gắn với task nào không?
- Có lưu số lần bị gián đoạn không?
- Có lưu trạng thái completed/abandoned không?
- Có nhiều preset Pomodoro không?

#### Về ôn tập

- Một note có thể sinh nhiều flashcard không?
- Flashcard có lịch ôn riêng không?
- Người dùng có trả lời flashcard không?
- Có lưu kết quả đúng/sai không?
- Có tính next review date không?

#### Về học nhóm

- Một LAN room có nhiều người tham gia không?
- Room có host không?
- Room có Pomodoro chung không?
- Room có file transfer không?
- Có lưu lịch sử hoạt động trong room không?

# 6. Kế hoạch chuẩn hóa 3NF

Khi thiết kế từng bảng, ta sẽ kiểm tra theo quy tắc sau.

## Bước 1: Kiểm tra 1NF

Mỗi ô chỉ chứa một giá trị.

Sai:

```
NoteId | Title | Tags
1      | Java  | java,network,socket
```

Đúng:

```
Notes
Tags
NoteTags
```

---

## Bước 2: Kiểm tra 2NF

Nếu bảng có khóa chính ghép, mọi thuộc tính phải phụ thuộc vào toàn bộ khóa.

Ví dụ bảng sai:

```
NoteTags(NoteId, TagId, TagName)
```

Khóa chính là:

```
(NoteId, TagId)
```

Nhưng `TagName` chỉ phụ thuộc vào `TagId`, không phụ thuộc vào `NoteId`.

Đúng:

```
Tags(TagId, TagName)
NoteTags(NoteId, TagId)
```

---

## Bước 3: Kiểm tra 3NF

Không có thuộc tính không khóa phụ thuộc vào thuộc tính không khóa khác.

Sai:

```
Tasks(TaskId, Title, PriorityId, PriorityName)
```

Vì:

```
TaskId → PriorityId
PriorityId → PriorityName
```

Có phụ thuộc bắc cầu:

```
TaskId → PriorityName
```

Đúng:

```
TaskPriorities(PriorityId, PriorityName)
Tasks(TaskId, PriorityId)
```

---

# 7. Nguyên tắc thiết kế SQL Server

## Đặt tên bảng

Nên dùng số nhiều hoặc số ít nhất quán. Mình khuyên dùng số nhiều:

```
Users
Notes
Tasks
FocusSessions
Flashcards
```

## Đặt tên khóa chính

```
UserId
NoteId
TaskId
FocusSessionId
```

## Đặt tên khóa ngoại

Giữ giống tên khóa chính của bảng cha:

```
UserId
NoteId
TaskId
```

## Kiểu dữ liệu khuyến nghị

```sql
INT IDENTITY(1,1)
BIGINT IDENTITY(1,1)
NVARCHAR(...)
DATETIME2
BIT
DECIMAL
UNIQUEIDENTIFIER
```

Với app học tập cá nhân, có thể dùng:

```sql
INT IDENTITY
```

Nếu muốn dễ đồng bộ nhiều thiết bị hoặc offline-first, có thể cân nhắc:

```sql
UNIQUEIDENTIFIER
```

Nhưng nếu làm đồ án SQL Server, `INT IDENTITY` sẽ dễ hơn.

## Các cột nên có ở hầu hết bảng chính

```
CreatedAt
UpdatedAt
DeletedAt nullable
IsDeleted
```

Hoặc nếu không làm soft delete thì chỉ cần:

```
CreatedAt
UpdatedAt
```

## Không nên lạm dụng soft delete

Với đồ án hoặc app vừa phải, chỉ nên soft delete các bảng quan trọng như:

- Notes
- Tasks
- Files
- Roadmaps
- Flashcards

Không nhất thiết soft delete lookup table như:

- TaskStatuses
- Priorities
- FileTypes

# 8. Các nhóm quan hệ quan trọng cần nghiên cứu kỹ

## 1. Note — Tag

Quan hệ nhiều-nhiều.


Bảng trung gian:

```
NoteTags
```

## 2. Note — Note

Một note có thể liên kết với nhiều note khác.

```
Notes N - N Notes

```


Ví dụ:

- Related
- Prerequisite
- Follow-up
- Reference

Một task có thể liên quan nhiều note, một note có thể liên quan nhiều task.

```
Tasks N - N Notes
```

Bảng trung gian:

```
TaskNotes
```

## 4. RoadmapTopic — Note / Task

Một topic trong roadmap có thể chứa nhiều note và nhiều task.

```
RoadmapTopicNotes
RoadmapTopicTasks
```

Một focus session có thể gắn với:

- Task
- Note
- Roadmap topic

```
FocusSessions
- TaskId nullable
- NoteId nullable
```

Về sau nếu muốn linh hoạt hơn, dùng bảng:

```
FocusSessionTargets
```

Nhưng như vậy phức tạp hơn.

## 6. Flashcard — Note

Một note có thể sinh nhiều flashcard.

```

```

## 7. LANRoom — User

Một room có nhiều member, một user có thể tham gia nhiều room.

```
LANRooms N - N Users
```

Bảng trung gian:

```
LANRoomMembers
```

# 10. Bộ câu hỏi nghiên cứu nghiệp vụ cần trả lời trước khi tạo bảng

Đây là checklist cực kỳ quan trọng.

## User

- Một user có thể có nhiều profile không?
- Có cần role admin/user không?
- Có cần tài khoản guest không?
- Có cần lưu thiết bị đăng nhập không?

## Note

- Note có thuộc đúng một subject không?
- Note có thể không thuộc subject nào không?
- Note có version history không?
- Note có thể bị archive không?
- Note có thể liên kết với nhiều note khác không?

## Tag

- Tag là riêng từng user hay dùng chung toàn hệ thống?
- Hai user có thể có tag cùng tên không?
- Tag có màu không?

Mình khuyên:

```
Tag nên thuộc về User.
```

Vì mỗi người có hệ thống tag riêng.

## Task

- Task có thể không có deadline không?
- Task có thể gắn nhiều note không?
- Task có recurring không?
- Task có subtask không?
- Task có category riêng hay dùng subject/category chung?

## Focus

- Focus session có bắt buộc gắn task không?
- Có cần lưu break session không?
- Có cần lưu số lần pause không?
- Có cần lưu interruption reason không?

## Review

- Flashcard có thuộc deck không?
- Deck có thuộc note không?
- Review schedule dùng thuật toán nào?
- Có lưu lịch sử từng lần review không?

## LAN

- Room có lưu vĩnh viễn không?
- User ngoài hệ thống có thể vào room không?
- Có cần lưu IP không?
- Có cần lưu file transfer thật không?
- Có cần chat không?

# 14. Đề xuất hướng nghiên cứu xuyên suốt

Mình sẽ định hướng database theo nguyên tắc sau:

> **Không thiết kế database theo UI trước, mà thiết kế theo nghiệp vụ học tập. UI chỉ là cách hiển thị dữ liệu.**
> 

Ví dụ:

UI có “Recent Notes”, nhưng database không cần bảng `RecentNotes`.

Vì recent notes có thể lấy bằng:

```sql
SELECT TOP 10 *
FROM Notes
WHERE UserId = @UserId
ORDER BY UpdatedAt DESC;
```

UI có “Today’s Focus”, nhưng database không cần bảng `TodaysFocus`.

Có thể lấy từ:

```
Tasks
FocusSessions
DailyGoals
```

UI có “Tags Cloud”, nhưng database không cần bảng `TagsCloud`.

Có thể tính từ:

```
Tags
NoteTags
TaskTags
```

UI có “Daily Streak”, nhưng có thể tính từ:

```
FocusSessions
StudyLogs
```

Chỉ khi cần tối ưu hiệu năng mới tạo bảng thống kê riêng.

# 15. Kết luận kế hoạch

Hướng đi đúng cho database của app bạn là:

```
User
→ Knowledge
→ Task
→ Focus
→ Review
→ Roadmap
→ Analytics
→ Collaboration
```

Và nguyên tắc thiết kế là:

```
Mỗi bảng lưu một loại thực thể.
Mỗi quan hệ nhiều-nhiều phải có bảng trung gian.
Mỗi dữ liệu phân loại nên có bảng lookup nếu cần quản lý rõ ràng.
Không lưu danh sách trong một cột.
Không lưu dữ liệu suy ra được nếu chưa cần thiết.
Không để thuộc tính không khóa phụ thuộc vào thuộc tính không khóa khác.
```

Tóm lại, database của bạn nên được nghiên cứu và xây dựng như một **hệ thống học tập cá nhân có khả năng mở rộng**, không chỉ là vài bảng note/task đơn giản.

Định hướng cuối cùng:

> **Cơ sở dữ liệu Focus-Node sẽ là một database SQL Server chuẩn 3NF, phục vụ việc quản lý tri thức, nhiệm vụ học tập, phiên tập trung, lộ trình học, ôn tập, tài liệu, thống kê tiến độ và học nhóm LAN. Thiết kế sẽ bắt đầu từ MVP khoảng 20–25 bảng, sau đó mở rộng lên 40–60+ bảng khi thêm roadmap, flashcard, review system và collaboration.**
